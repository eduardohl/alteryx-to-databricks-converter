Who Is This For?

Data Engineers

Converting Alteryx pipelines to PySpark/SQL for Databricks lakehouse.

Migration Leads

Planning portfolio-wide migration: effort estimates, wave grouping, progress tracking.

Analysts

Getting runnable Databricks code from existing workflows without writing PySpark by hand.

When NOT to use a2d: If the workflow is purely spatial, predictive/ML, or R-based, manual rewrite is faster. a2d will flag these honestly with # TODO markers.

The Problem

Organizations use Alteryx Designer to build hundreds of visual data pipelines.

Migrating to Databricks usually means:

  • Reading every workflow manually — opaque XML under the visual canvas
  • Rewriting each one from scratch in PySpark or SQL
  • Weeks per workflow × hundreds of workflows = months of engineering effort

a2d does not require Alteryx to be installed. It reads the .yxmd XML directly.

Phase 1 — Parse

lxml reads XML, maps plugin names via PLUGIN_NAME_MAP, builds a WorkflowDAG of typed ParsedNode objects.

Phase 2 — Convert & Generate

Each node is converted to a typed IR node. Expressions go through a full recursive-descent compiler. The DAG is walked in topological order to emit code.

Four Output Formats

Every conversion emits all four formats by default into per-format subdirectories (output/pyspark/, output/dlt/, output/sql/, output/lakeflow/). The -f flag is a comma-separated filter (e.g. -f pyspark,sql). Each format also produces a Workflow JSON for Databricks Jobs orchestration.

Output: PySpark

Interactive notebook with DataFrame API

# Databricks notebook source
# Converted from: sales_pipeline.yxmd
# Format: pyspark | Nodes: 8 | Confidence: 91%

from pyspark.sql import functions as F

# Step 1: Input Data (CSV)
df_1 = spark.read.format("csv").option("header", True).load("/data/sales.csv")

# Step 3: Filter
df_3_true = df_1.filter(F.col("status") == "active")
df_3_false = df_1.filter(~(F.col("status") == "active"))

# Step 4: Formula
df_4 = df_3_true.withColumn("total", F.col("price") * F.col("qty"))

# Step 5: Summarize
df_5 = df_4.groupBy("region").agg(
    F.sum("total").alias("revenue"),
    F.count("*").alias("orders")
)

# Step 7: Output Data
df_5.write.format("delta").mode("overwrite").saveAsTable("main.default.sales_summary")

Output: Spark Declarative Pipelines (DLT)

Declarative @dlt.table functions — each step is a managed table

import dlt
from pyspark.sql import functions as F

@dlt.table(comment="Step 1: Input Data (CSV)")
def step_1_input_data():
    return spark.read.format("csv").option("header", True).load("/data/sales.csv")

@dlt.table(comment="Step 3: Filter (True branch)")
def step_3_filter_true():
    return dlt.read("step_1_input_data").filter(F.col("status") == "active")

@dlt.table(comment="Step 4: Formula")
def step_4_formula():
    return dlt.read("step_3_filter_true").withColumn(
        "total", F.col("price") * F.col("qty")
    )

@dlt.table(comment="Step 5: Summarize")
def step_5_summarize():
    return dlt.read("step_4_formula").groupBy("region").agg(
        F.sum("total").alias("revenue"),
        F.count("*").alias("orders")
    )

Output: Databricks SQL

Single CTE chain — one WITH block, each step is a named CTE

-- Migrated from: sales_pipeline.yxmd
-- Generated by: a2d | Confidence: 91%

WITH step_1_input_data AS (
  SELECT * FROM csv.`/data/sales.csv`
),
step_3_filter AS (
  SELECT * FROM step_1_input_data WHERE `status` = 'active'
),
step_4_formula AS (
  SELECT *, `price` * `qty` AS `total` FROM step_3_filter
),
step_5_summarize AS (
  SELECT `region`,
         SUM(`total`) AS `revenue`,
         COUNT(*) AS `orders`
  FROM step_4_formula
  GROUP BY `region`
)
SELECT * FROM step_5_summarize;

Output: Lakeflow Designer

Individual DDL statements with LIVE. prefix on upstream refs

-- Lakeflow Designer pipeline: sales_pipeline
-- Format: Lakeflow Declarative Pipelines (LDP) SQL

-- Step 1: Input Data (CSV) → STREAMING TABLE (file-based source)
CREATE OR REFRESH STREAMING TABLE step_1_input_data AS
SELECT * FROM csv.`/data/sales.csv`;

-- Step 3: Filter → MATERIALIZED VIEW
CREATE OR REFRESH MATERIALIZED VIEW step_3_filter_true AS
SELECT * FROM LIVE.step_1_input_data WHERE `status` = 'active';

CREATE OR REFRESH MATERIALIZED VIEW step_3_filter_false AS
SELECT * FROM LIVE.step_1_input_data WHERE NOT (`status` = 'active');

-- Step 5: Summarize → MATERIALIZED VIEW
CREATE OR REFRESH MATERIALIZED VIEW step_5_summarize AS
SELECT `region`, SUM(`total`) AS `revenue`, COUNT(*) AS `orders`
FROM LIVE.step_4_formula GROUP BY `region`;

Also produces _lakeflow_pipeline.json with catalog, target, cluster config.

CLI deploy-readiness banner & categorized warnings

After every conversion the CLI prints a 3-tier banner mirroring the UI Convert page:

  • ✓ Ready to deploy — clean coverage, no blockers
  • ⚠ Needs review — warnings present but non-blocking
  • ✗ Cannot deploy as-is — blockers present (unsupported tools, missing generators, parser failures)

Warnings are grouped into Cannot convert, Manual review needed, Graph structure note, and Other instead of a flat dump. The same rules drive the API deploy_status field and the React Convert page, so CLI and UI agree byte-for-byte.

Pipeline Architecture


  .yxmd XML
       |
  [Parser]  ──── lxml reads XML → WorkflowDAG of ParsedNode objects
       |         Maps Alteryx plugin names via PLUGIN_NAME_MAP (157 entries)
       |
  [Converters]  ──── 62 tool-specific converters (ParsedNode → IRNode)
       |              Expression engine: tokenizer → parser → AST → translator
       |
  [Generator]  ──── Walks DAG in topological order
       |             Emits format-specific code (PySpark / DLT / SQL / Lakeflow)
       |
  [Post-processing]  ──── Syntax validation (ast.parse)
       |                   Confidence score (5 dimensions)
       |                   Complexity analysis (7 factors)
       |
  Output  ──── .py / .sql + workflow.json + optional: DDL, DAB, tests, audit CSV

Orchestrated by pipeline.py. Single entry point: ConversionPipeline.run()

Converter Registry

@ConverterRegistry.register("Filter", "preparation")
class FilterConverter(ToolConverter):
    def convert(self, parsed: ParsedNode, dag: WorkflowDAG) -> FilterNode:
        expression = safe_get_nested(
            parsed.config, "Expression", "Simple", default=""
        )
        return FilterNode(
            node_id=parsed.node_id,
            expression=expression,
            ...
        )

Expression Edge Cases

141 functions + special constructs

AlteryxPySparkNotes
[Row-1:Amount] F.lag("Amount", 1).over(window) Row references → window functions
Switch([S], "?", "A", "Active", "I", "Inactive") F.when(col == "A", "Active").when(col == "I", "Inactive").otherwise("?") Variable-arity dispatch
DateTimeAdd([Date], 30, "days") F.date_add(F.col("Date"), 30) Unit-aware: days, months, years
Null() F.lit(None) Null-as-function, not keyword
IN([Status], "A", "B", "C") F.col("Status").isin("A", "B", "C") Varargs → .isin()
When an expression can't be translated, a safe placeholder is emitted: F.lit(True) for filters, F.lit(None) for formulas — with the original expression in a # TODO comment.

What happens when things can't be converted

ScenarioWhat a2d Does

Unsupported tool (e.g. Spatial)Recognized, emits # TODO stub, passes data through
Unknown tool (not in registry)Emits # TODO: Unsupported tool <name>, passes data through
Empty expression fieldRaises BaseTranslationError, caught by generator

Guarantee: Generated code is always syntactically valid Python/SQL. It may contain TODOs, but it will parse.

Automatic Optimizations

BehaviorWhat It Does
Fan-out cachingWhen a node feeds 2+ downstream steps, .cache() is inserted automatically to prevent recomputation
Passthrough eliminationNo-op tools (AutoField, Browse, BlockUntilDone) don't emit df_N = df_M — downstream code references the original variable
Syntax validationast.parse() runs on every generated .py file; Databricks magic lines are stripped first
withColumns batchingMultiFieldFormula emits one .withColumns({...}) instead of N separate .withColumn() calls
Chain flatteningSelect renames+drops chained as one expression: inp.withColumnRenamed(...).drop(...)
Literal wrappingBare number constants (e.g. 500) automatically wrapped in F.lit(500)

  • Example — sample fix or Databricks equivalent

Day-in-the-Life Walkthrough

Converting 50 workflows in a week

Monday: Assess — a2d analyze workflows/ -o report/ --format both --complexity

Review the HTML report. Identify "Low" workflows (quick wins) and "High" workflows (complex). Create connection mapping YAML.

Tuesday-Wednesday: Batch convert — a2d convert workflows/ -o output/ --batch --report-format html --connection-map map.yml --generate-ddl --generate-dab

Review batch report. Search output files for # TODO. Fill in ODBC connections, Excel paths.

Thursday: Validate — a2d validate output/*.py

Upload notebooks to Databricks. Run against sample data. Compare row counts between original and converted.

Friday: Deploy quick wins — upload Workflow JSON, run as Databricks Job. Or use databricks bundle deploy with the generated DAB config.

Reading Reports

Key Metrics

  • Confidence — 80%+ run with minor edits, <50% significant work
  • Complexity — maps to effort hours (2h – 40h)
  • Coverage — % of nodes with converters
  • Warning count — fewer is better; check enriched hints

Report Formats

  • HTML — visual, shareable, confidence column, recommendations section
  • JSON — programmatic, includes enriched_warnings with hint/category per warning
  • JSONL — one record per line for log ingestion

Tips & Tricks

  1. Search output for # TODO and # WARNING first
  2. Check confidence — 80%+ means minimal manual work
  3. Use --expression-audit to find failed translations instantly
  4. Use --performance-hints for Spark-specific optimizations
  5. Run a2d validate before uploading to Databricks
  6. Map connections first with --connection-map to reduce TODOs
  1. Start with high-confidence workflows for quick wins
  2. Use --generate-ddl + --connection-map to reduce manual TODO count
  3. Use --generate-dab for Databricks Asset Bundle deployment
  4. Use --comments for team review — every step is documented
  5. Run a2d analyze --complexity first for large portfolios — prioritize by effort
  6. Check enriched warnings in batch HTML reports for actionable hints

What a2d Does NOT Do

Honest limitations

LimitationWhyWorkaround
Spatial analysisNo PySpark equivalent for spatial primitivesUse Sedona / Mosaic / H3; a2d emits TODO stubs
ML model trainingAlteryx predictive tools use R/Python ML under the hoodRebuild with MLflow + scikit-learn
Iterative macrosRequire loop semantics not expressible in a DAG walkRewrite as Lakeflow Jobs with loops
Semantic equivalencea2d validates syntax, not business logicCompare row counts manually in Databricks
Custom third-party toolsUnknown XML schemaAdd a custom converter (see Extending a2d)
Data type inferenceAlteryx types don't always map 1:1 to SparkReview generated schema, add explicit casts
Runtime testinga2d doesn't execute the generated codeRun in Databricks, validate with test scaffold

Extending a2d

Adding a New Converter

  1. Create file in converters/<category>/
  2. Add IR node in ir/nodes.py
  3. Implement with @ConverterRegistry.register
  4. Add plugin mapping in schema.py
  5. Add visitor in generators (Lakeflow inherits SQL)
  6. Add unit test

Adding an Expression Function

# In expressions/functions.py:
FUNCTION_REGISTRY["MyFunc"] = (
    lambda args: f"my_udf({args[0]})",
    1,  # expected arg count
)

141 functions already supported. Add new ones to FUNCTION_REGISTRY with a PySpark translation lambda.
